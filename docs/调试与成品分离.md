# 调试页 / 成品页分离（架构约定）

> 正式实施前定死的工程约定。UI/动效会反复调，必须从结构上把"调试"和"成品"隔开，
> 否则两个后果：① 调试代码（控制面板、帧率、网格、慢放）漏进成品包；② 调试页和成品页
> 各改各的，参数/效果漂移对不上。

---

## 0. 一句话原则

**效果（effect）是唯一真源。调试页（Lab）和成品页（Gallery）都只是"消费"同一个 effect 模块——
绝不在两边各写一份效果代码。** 调试专属的东西（lil-gui 面板、stats 帧率、采样点可视化、慢放/单步）
只存在于 Lab，永不进成品构建。

---

## 1. 三层结构

```
src/
  effects/            ← 唯一真源：纯效果模块，参数化，零调试代码、零业务逻辑
    textParticles/    每个 effect 一个目录，实现统一的 EffectModule 契约
    spellVoice/
    scratchReveal/
    aurora/
  lab/                ← 仅调试：每个 effect 一个隔离路由 + 控制面板 + 帧率 + 辅助层
  gallery/            ← 成品展厅：每个 section 用「锁定的预设参数」包一个 effect，无任何调试 UI
  shared/             ← 公用（滚动引擎 Lenis/GSAP、主题、工具）
```

- `effects/`：和 UI 框架解耦，给个 `container` 和 `params` 就能跑，不知道自己被 Lab 还是 Gallery 用。
- `lab/`：开发时把玩、拨参数、看性能的地方。
- `gallery/`：评委看到的最终页，只读冻结好的参数。

---

## 2. EffectModule 契约（关键——两边共用的接口）

```ts
// 每个效果都实现这个，Lab 和 Gallery 都按它来消费
export interface EffectHandle<P> {
  update(params: Partial<P>): void   // Lab 拨参数 → 实时生效
  resize(): void
  reset(): void
  getParams(): P                     // Lab「导出参数」→ 冻结成 Gallery 的 preset
  destroy(): void
}

export interface EffectModule<P> {
  id: string
  defaultParams: P
  presets?: Record<string, P>        // 例如按 genre：detective / fantasy / scifi
  schema: ParamSchema<P>             // 描述每个参数的 min/max/step/类型/颜色
  mount(container: HTMLElement, params: P): EffectHandle<P>
}
```

**一份 `schema` 复用三处**（这是这套约定最值的地方）：
1. **Lab**：自动生成 lil-gui 控制面板（拿 min/max/step 直接渲染滑块/取色器）。
2. **Gallery**：成品加载参数时做校验，挡掉非法值。
3. **Spellbook Compiler（AI）**：约束 LLM 输出的参数必须落在 schema 内（结构化输出的 JSON Schema）。

## 3. 工作流：调好 → 冻结 → 用（正好对上"等美术资源再微调"）

```
美术资源到位
   ↓
在 Lab 里换素材 + 拨参数，直到满意（实时预览，不碰 Gallery）
   ↓
点「Copy params JSON」
   ↓
粘进 gallery 对应 section 的 preset
   ↓
成品自动更新，Gallery 布局零改动
```

> 微调永远发生在 Lab；Gallery 只吃冻结结果。所以"等美术再微调"= 等美术 → 进 Lab → 冻结，
> 不会动到成品结构。

---

## 4. 工程实现（Vite 多入口，构建层面隔离）

- `index.html` → **Gallery**（production 唯一入口）
- `lab.html` → **Lab**（开发用）
- `vite build` 只产出 Gallery；Lab 走 `npm run lab` / dev，**默认不打进生产包**
  （或用 `import.meta.env.DEV` / 路由门控彻底排除）。
- `lil-gui`、`stats.js` 只在 `lab/` 里 import → tree-shaking 保证它们不进成品。

`package.json` 脚本约定：

```jsonc
{
  "dev":   "vite",                       // 开发（含 Lab）
  "lab":   "vite --open /lab.html",      // 直接进调试台
  "build": "vite build",                 // 只构建 Gallery 成品
  "preview": "vite preview"
}
```

---

## 5. 调试页应具备的能力（实施时照这个清单做 Lab）

- 实时参数面板（lil-gui，由 effect 的 `schema` 自动生成）
- FPS / 帧时（stats.js）+ 移动端 **DPR 档位切换**（1x/2x）
- **reset / 暂停 / 单步 / 慢放（0.1x–2x）**
- 可视化辅助层开关：粒子**采样点**、bounds、安全区、网格、触控热区
- **「Copy params JSON」**「截一帧 PNG」
- **视口模拟**：phone / pad 尺寸一键切换（成品是移动端优先）
- 多模态**兜底开关**：语音/麦克风的 demo-mode 直接在 Lab 里测稳

---

## 6. 当前 `spikes/` 就是 Lab 的胚胎

`spikes/` 里每个独立 HTML（text-particles / spell-voice / knock-mic）= 一个 effect 的隔离调试页雏形。
正式实施时：把效果逻辑抽进 `effects/<id>/`，参数抽成 `params.ts` + `schema`，接上 lil-gui，
就升级成正式 `lab/`。**不丢弃、平滑迁移。**

---

## 7. 红线（评审/自检时按这个查）

- ❌ Gallery 里出现 lil-gui / stats / `console.log` 调参 / 写死的 magic number → 不合规
- ❌ 同一个效果在 spikes 和 gallery 各有一份实现 → 必须合并到 `effects/`
- ✅ 改一个效果只改 `effects/<id>/`，Lab 和 Gallery 同时受益
- ✅ 成品参数全部来自 `presets` / 冻结 JSON，不在组件里硬编码
